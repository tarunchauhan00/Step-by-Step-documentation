    How It Works¶

    This section describes the internal architecture, the main components, and how data flows through the app.

    1. High-Level Architecture¶

    1. Navbar (src/components/Navbar.tsx)

    2. Present on all private pages; contains links to Processes, Add Process, Form Builder, etc.

    3. ProcessCard (src/components/ProcessCard.tsx)

    4. Displays a Process’s title, description snippet, number of steps, created date.

    5. Click → /process/:processId/steps.

    6. StepCard (src/components/StepCard.tsx)

    7. Shows Step title, due date, assigned doer, form link.

    8. Click → edit/view that Step.

    9. PrivateRoute (src/components/PrivateRoute.tsx)

    10. Guards every route that requires authentication.

    2. Supabase Schema & Client¶

    Below is a suggested table structure. The React front-end assumes these tables exist:

    Table: users¶

    • id (UUID, primary key)
    • email (text)
    • full_name (text)
    • role (enum: 'owner' | 'doer')
    • created_at, updated_at

    Table: processes¶

    • id (UUID, primary key)
    • title (text)
    • description (text)
    • owner_id (UUID, references users.id)
    • created_at, updated_at

    Table: steps¶

    • id (UUID, primary key)
    • process_id (UUID, references processes.id)
    • title (text)
    • description (text)
    • due_date (date)
    • doer_id (UUID, references users.id)
    • form_schema_id (UUID, references form_schemas.id, nullable)
    • google_form_url (text, nullable)
    • status (enum: 'pending' | 'completed' | 'overdue')
    • created_at, updated_at

    Table: form_schemas¶

    • id (UUID, primary key)
    • name (text)
    • description (text)
    • schema (JSONB)
    • created_at, updated_at

    Table: form_responses¶

    • id (UUID, primary key)
    • form_id (UUID, references form_schemas.id) OR step_id (UUID, references steps.id)
    • doer_id (UUID, references users.id)
    • response_data (JSONB)
    • submitted_at (timestamp)

    3. How Each Page Works¶

    3.1. AddProcess (src/pages/AddProcess.tsx)¶

    • Purpose: Create or edit a Process.
    • Inputs: Title, Description, Owner (current user), optional “Default Deadline.”
    • Flow:
    • If processId exists → fetch that process and prefill fields.
    • On “Save” → supabase.from('processes').insert(...) or update(...).
    • Redirect to /processes.

    3.2. ProcessList (src/pages/ProcessList.tsx)¶

    • Purpose: List all Processes owned by current user.
    • Features: Search, filter, “Add Process.”
    • Click a ProcessCard → /process/:processId/steps.

    3.3. AddStep (src/pages/AddStep.tsx)¶

    • Purpose: Create or edit a Step.
    • Inputs: Step Title, Description, Due Date, Assigned Doer, Google Form URL (optional), Custom Form ID (optional).
    • Flow:
    • If stepId exists → fetch and prefill.
    • On “Save” → supabase.from('steps').insert(...) or update(...).
    • Redirect → /process/:processId/steps.
